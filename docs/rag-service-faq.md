# RAG Service 面试 FAQ

## 基础概念

### Q1: 什么是 RAG？为什么需要 RAG？

RAG（Retrieval-Augmented Generation）是**检索增强生成**。核心思路是：在 LLM 回答问题之前，先从外部知识库中检索相关文档片段，作为上下文提供给模型，让模型基于检索到的内容来生成答案。

需要 RAG 的原因：
- **解决知识截止问题**：LLM 训练数据有截止日期，新知识不知道
- **减少幻觉**：模型凭空编造的概率降低，因为有参考文本
- **可溯源**：答案可以引用原文，用户可以验证
- **私有数据**：企业文档、内部知识库不需要重新训练模型

### Q2: 稠密向量和稀疏向量的区别是什么？

| 维度 | 稠密向量 | 稀疏向量 |
|------|---------|---------|
| 表示方式 | 固定长度的浮点数数组（如 1024 维） | 高维空间中大部分为 0 的向量（如 65536 维） |
| 语义能力 | 强，能捕捉"汽车"和"车辆"的相似性 | 弱，本质是关键词匹配 |
| 精确匹配 | 弱，同义词可能漂移 | 强，"HNSW"就是"HNSW" |
| 存储开销 | 固定大小 | 随非零元素数量线性增长 |

**为什么混合检索二者都要？**

- 只用稠密向量 → 检索专有名词、产品型号、代码片段时效果差
- 只用稀疏向量 → 同义词/近义词/不同表述会漏召
- 混合 = 语义理解（稠密）+ 精确命中（稀疏），互补短板

### Q3: 什么是分块（chunking）？分块大小如何影响检索质量？

分块是把长文档切分成小片段的过程，因为：
- LLM 上下文窗口有限
- 细粒度检索：整篇文章搜不如搜段落精准
- 向量模型有最大输入长度限制

**chunk 太大的问题：**
- 一个 chunk 包含多个主题，检索不精准
- 浪费上下文窗口（塞了很多无关内容）
- 超出向量模型的 max tokens 会被截断

**chunk 太小的问题：**
- 缺少上下文，单独一段话可能语义不全
- 检索命中但 LLM 无法理解
- 索引量暴增，搜索效率下降

一般来说，chunk 大小在 256~1024 tokens 之间，取决于文档类型和应用场景。

## 项目架构

### Q4: 整个 rag-service 的核心 pipeline 是什么？

```
输入文档 → 自适应分块 → 向量化(稠密+稀疏) → FAISS 建索引 → 
用户 query → 向量化 → 混合检索 → [rerank] → 返回 top-k 结果
```

四个核心模块：
1. **Chunker（分块器）** — 把原始文本切成语义完整的片段
2. **Embedder（向量化器）** — 用 bge-m3 模型把文本转为稠密+稀疏向量
3. **Indexer（索引器）** — 用 FAISS HNSW 存稠密向量，scipy CSR 存稀疏向量
4. **Retriever（检索器）** — 执行混合检索，融合分数，可选 rerank

### Q5: 什么是 HNSW？为什么选择它？

HNSW（Hierarchical Navigable Small World）是一种**近似最近邻搜索**算法，用多层的图结构来加速向量检索。

选择原因：
- **速度快**：百万级向量上也能达到毫秒级响应
- **精度高**：接近暴力搜索（KNN）的召回率
- **全内存**：所有向量加载到内存，无磁盘 I/O 瓶颈
- **FAISS 内置**：无需额外依赖

代价是建索引慢（需要构建图结构），且增量添加会逐渐降低搜索质量，这是你的代码中有一个 warn log 的原因。

### Q6: 什么是多租户（multi-tenant）？你的项目怎么做的？

多租户指同一个服务为多个独立的用户/团队/项目提供隔离的知识库。每个租户的数据互不可见。

你的做法是最简单的**目录隔离**：
```
index_dir/
├── tenant_a/
│   ├── dense.faiss
│   ├── sparse.npz
│   └── chunks.jsonl
└── tenant_b/
    ├── dense.faiss
    ├── sparse.npz
    └── chunks.jsonl
```

优缺点：
- ✅ 实现简单，无外部依赖
- ✅ 删除租户就是删文件夹
- ❌ 所有租户在同一进程，一个 OOM 全挂
- ❌ 搜索热门租户时冷门租户不受影响（实际会，因为单线程）
- ❌ 不能跨租户搜索

### Q7: 你们的 reranker 在 pipeline 中处于什么位置？为什么需要它？

位置在**混合检索之后、返回最终结果之前**：

```
混合检索(recall_top_k=50) → reranker 精排 → 取 top_k 返回
```

为什么需要：
- 向量检索是"近似"的，粗召回来的结果里可能混入不太相关的
- reranker 是 cross-encoder，把 query 和候选文档 pair 送入模型，打分更准确
- 计算成本高（只能对少量候选用），所以放在**第二阶段精排**

## 工程实践

### Q8: 并发写入同一个知识库会怎样？怎么处理的？

使用 **filelock**（文件锁）机制：
- 每个知识库目录下有一个 `.lock` 文件
- 获取锁时设置 5 秒超时
- 超时获取不到就抛 `IndexLockedError`

所以如果两个请求同时写入同一个 tenant，第二个会拿到错误而不是数据损坏。但对于 REST API 来说，这个错误会变成 HTTP 400，前端没有自动重试。

### Q9: 索引是增量追加还是全量重建？

**两种都支持**：
- 首次写入 → `build()`，创建新索引
- 后续写入 → `add()`，向已有索引追加

`add()` 有潜在问题：HNSW 是一个图结构，不断追加数据会让图的连接质量下降。代码里有一个预警：当增量添加超过总量 20% 时，建议全量重建。

### Q10: 这个项目有哪些地方可以改进？

**功能性：**
- 缺少 query 理解/改写（虽然预留了 `query_rewriter` 接口）
- 不支持文档更新（update），只能删了重建
- 不支持标量过滤（如只搜某个来源的文档）

**可靠性：**
- 数据存在单机磁盘上，没有备份/容错
- `text.find()` 定位段落位置可能有 bug（重复文本会匹配错）
- 没有监控/metrics

**性能：**
- 模型加载是懒加载，但第一次请求会卡住
- 所有索引全在内存，没有分片
- 没有连接池/请求排队

**测试：**
- 测试覆盖了核心模块，但缺少集成测试（端到端跑一次真实模型加载）
- 没有性能测试/基准测试

## 技术选型

### Q11: 为什么用 bge-m3 这个模型？

bge-m3（BAAI 出品）是目前中文+英文混合场景下 F1 领先的 embedding 模型，特点是：
- 同时输出稠密向量（1024维）和稀疏向量（65536维）
- 支持多种语言（中文、英文等）
- MRL 机制：部分向量可以截断用，不影响已有索引

### Q12: 为什么 scipy CSR 存稀疏向量？

CSR（Compressed Sparse Row）是存储稀疏矩阵的标准格式：
- 只存非零元素，节省内存（65536 维里通常只有几十个非零）
- 矩阵乘法速度快（`q @ matrix.T` 就是一次稀疏点积）
- scipy 生态成熟，能和 numpy/faiss 互操作

### Q13: FastAPI 在这项目里起什么作用？

FastAPI 是可选的 HTTP 外壳：
- 让 Python 库变成一个可通过 REST API 调用的服务
- 自带 Swagger UI 方便调试
- 通过 lifespan 管理 RAGService 生命周期
- 挂载了一个前端演示页面（static 目录）

核心逻辑和 HTTP 层解耦：你可以 `pip install rag-service` 纯当 Python 库用，也可以 `python server.py` 当服务用。

## 面试追问

### Q14: RAG 和 Fine-tuning 有什么区别？什么时候用 RAG 什么时候用 Fine-tuning？

| 维度 | RAG | Fine-tuning |
|------|-----|-------------|
| 知识更新 | 秒级（换索引就行） | 天~周级（重新训练） |
| 长尾知识 | 效果好（可以直接搜到） | 容易遗忘 |
| 推理成本 | 更高（检索 + 长上下文） | 更低（模型已记住） |
| 幻觉 | 更低（有参考文本） | 可能记住错误知识 |
| 实现复杂度 | 简单（无需训练） | 需要训练 pipeline |

**选 RAG 的场景**：企业知识库、实时新闻问答、产品文档、法律法规等频繁更新的场景。

**选 Fine-tuning 的场景**：固定格式输出（如信息抽取）、让模型模仿特定风格/语气、降低推理成本。

实际项目中常**两者结合**：用 RAG 提供事实知识，用 Fine-tuning 教会模型如何使用这些信息。

### Q15: 如何评估 RAG 系统的检索质量？

常用指标：
- **Recall@k**：正确答案在前 k 个结果中的比例
- **MRR（Mean Reciprocal Rank）**：第一个正确答案排得有多靠前
- **NDCG@k**：考虑排序位置的加权指标

评估方式：
- 人工标注：构造 query-相关文档对
- 公开 benchmark：MTEB、BEIR 等
- 端到端评估：生成的答案质量才是最终指标（使用 LLM-as-Judge）
